# UART-протокол кондиционера DEXP

Этот документ описывает текущее состояние декодирования UART-протокола между
штатным Wi-Fi модулем и внутренним блоком кондиционера DEXP.

Протокол декодирован по UART-дампам, снятым логическим анализатором, и
сопоставлен с изменениями состояния в Home Assistant и видео нажатий кнопок
пульта.

## UART

- Скорость: `9600`
- Биты данных: `8`
- Четность: нет
- Стоп-биты: `1`
- Пины логического анализатора:
  - `D11`: кондиционер -> Wi-Fi модуль
  - `D12`: Wi-Fi модуль -> кондиционер

## Формат кадра

Все известные кадры начинаются с `AA`.

```text
AA LEN ... CHECKSUM
```

`LEN` - количество байт, оставшихся после байта `LEN`, включая байт
контрольной суммы.

Контрольная сумма:

```text
sum(all bytes before checksum) & 0xFF
```

Пример:

```text
AA 02 01 AD
AA + 02 + 01 = AD
```

## Опрос

Wi-Fi модуль отправляет периодический запрос примерно каждые 5 секунд:

```text
AA 02 01 AD
```

Кондиционер отвечает кадром состояния:

```text
AA 14 <19-byte payload> CHECKSUM
```

Для кадров `AA 14` индексы ниже указаны как zero-based индексы внутри
`payload`, без учета `AA`, `LEN` и `CHECKSUM`.

## Handshake при подаче питания

Дамп, записанный с момента подачи питания, показал короткий стартовый handshake
до начала обычного опроса:

```text
wifi_to_ac: AA 03 03 03 B3
ac_to_wifi: AA 03 03 03 B3
```

Этот обмен наблюдался дважды, с интервалом около 1.2 секунды. В том же дампе
обычный опрос `AA 02 01 AD` начался после этого, примерно через 14 секунд от
старта захвата.

Стартовый кадр использует то же правило контрольной суммы:

```text
AA + 03 + 03 + 03 = B3
```

В этом power-on дампе до начала обычного опроса не наблюдалось командного
кадра `AA 14` от Wi-Fi модуля к кондиционеру.

## Командные кадры

Wi-Fi модуль также отправляет кадры `AA 14`, когда управляет кондиционером.

Наблюдаемые командные кадры используют ту же длину и то же правило контрольной
суммы, что и кадры состояния, но `payload` начинается с `02`:

```text
AA 14 <19-byte command payload> CHECKSUM
```

Пример командного кадра:

```text
AA 14 02 01 00 06 18 01 10 03 17 17 00 00 00 00 00 16 38 00 00 6F
```

Payload:

```text
02 01 00 06 18 01 10 03 17 17 00 00 00 00 00 16 38 00 00
```

Известные поля командного `payload`:

| Индекс payload | Известное значение |
| ---: | --- |
| `0` | Тип кадра: `02` - команда |
| `1` | Желаемое питание |
| `2` | Неизвестно, наблюдалось `00` |
| `3` | Неизвестно, кандидат на display/light, наблюдалось `06` |
| `4` | Неизвестно в командных кадрах, наблюдалось `18` или `00` |
| `5` | Желаемый режим HVAC |
| `6` | Битовая маска специальных функций |
| `7` | Желаемая скорость вентилятора |
| `8` | Желаемая целевая температура |
| `9` | Копия текущей температуры или `00` |
| `10..14` | Наблюдалось `00` |
| `15` | Час timestamp команды, бинарное значение часа, например `01` в 01:xx |
| `16` | Минута timestamp команды, бинарное значение минуты, например `0C` = 12, `11` = 17, `16` = 22, `17` = 23 |
| `17..18` | Наблюдалось `00` |

Кондиционер отвечает на команду кадром `AA 14`, у которого `payload` тоже
начинается с `02`. Эти ack/response кадры зеркалируют желаемые поля состояния,
но обычно зануляют часть несущественных полей, например `payload[4]`,
`payload[9]` и `payload[15..18]`.

Пример команды и немедленного ответа кондиционера:

```text
wifi_to_ac: AA 14 02 01 00 06 18 01 10 03 17 17 00 00 00 00 00 16 38 00 00 6F
ac_to_wifi: AA 14 02 01 00 06 00 01 10 03 17 00 00 00 00 00 00 00 00 00 00 ...
```

Следующий обычный ответ на poll затем возвращает кадр состояния с
`payload[0] == 01`.

Чистые HA-driven дампы показывают такой рабочий шаблон команды:

```text
02 POWER 00 06 18 MODE SPECIAL FAN TARGET CURRENT 00 00 00 00 00 01 XX 00 00
```

Наблюдаемые значения `payload[15..16]` совпадают с локальным временем команды:

| Время дампа | Наблюдаемое `payload[15..16]` | Декодированное время |
| --- | --- | --- |
| 01:12 | `01 0C` | 01:12 |
| 01:13 | `01 0D` | 01:13 |
| 01:17 | `01 11` | 01:17 |
| 01:22 | `01 16` | 01:22 |
| 01:23 | `01 17` | 01:23 |

Это указывает на то, что `payload[15]` - час, а `payload[16]` - минута,
закодированные обычными бинарными значениями и отображаемые в логах как hex
байты. Это не BCD: `0x16` означает десятичную минуту 22, а не минуту 16.

Для первого генератора команд безопаснее может быть переиспользовать последние
наблюдавшиеся context-байты штатного Wi-Fi модуля, но лучшая текущая гипотеза
для проверки на железе - генерировать текущие локальные час и минуту.

## Payload состояния

Пример кадра состояния:

```text
AA 14 01 01 00 06 18 01 10 00 18 18 00 00 00 00 00 00 00 00 00 1F
```

Payload:

```text
01 01 00 06 18 01 10 00 18 18 00 00 00 00 00 00 00 00 00
```

| Индекс payload | Известное значение |
| ---: | --- |
| `0` | Тип кадра: `01` - обычное состояние, `02` - ack/response на команду |
| `1` | Питание: `00` выключено, `01` включено |
| `2` | Неизвестно, обычно `00` |
| `3` | Неизвестно, кандидат на display/light: наблюдалось `06` normal, `00` alternate/off |
| `4` | Флаги/состояние swing: наблюдалось `18` normal/off, `1A` horizontal swing |
| `5` | Режим HVAC |
| `6` | Битовая маска специальных функций |
| `7` | Скорость вентилятора |
| `8` | Целевая температура |
| `9` | Текущая температура |
| `10..18` | Неизвестно, в обычных кадрах состояния обычно `00` |

## Питание

Хранится в `payload[1]`.

| Значение | Смысл |
| --- | --- |
| `00` | Выключено |
| `01` | Включено |

## Режим HVAC

Хранится в `payload[5]`.

| Значение | Смысл |
| --- | --- |
| `00` | Auto |
| `01` | Cool |
| `02` | Dry |
| `03` | Heat |
| `04` | Fan only |

## Скорость вентилятора

Хранится в `payload[7]`.

| Значение | Смысл |
| --- | --- |
| `00` | Auto |
| `01` | Low |
| `02` | Middle |
| `03` | High |

## Температура

Целевая температура хранится в `payload[8]`.

Сырой байт равен температуре в градусах Цельсия:

| Диапазон raw | Смысл |
| --- | --- |
| `10..20` | `16..32` градусов C |

Примеры:

| Значение | Температура |
| --- | ---: |
| `10` | 16 C |
| `18` | 24 C |
| `20` | 32 C |

Текущая температура хранится в `payload[9]`.

Сырой байт также равен температуре в градусах Цельсия.

Примеры:

| Значение | Температура |
| --- | ---: |
| `16` | 22 C |
| `17` | 23 C |
| `18` | 24 C |
| `19` | 25 C |

В некоторых ack/command-response кадрах текущая температура может быть `00`.

## Специальные функции

Специальные функции хранятся в `payload[6]`.

Это поле является битовой маской с базовым/нормальным marker-битом `0x10`.

```text
payload[6] = 0x10 | flags
```

Известные флаги:

| Бит | Маска | Функция |
| ---: | --- | --- |
| 0 | `0x01` | Sleep / сон |
| 1 | `0x02` | Ionizer / ионизатор |
| 2 | `0x04` | Mute / quiet / тихий режим |
| 3 | `0x08` | Strong / turbo |

Наблюдаемые значения:

| Значение | Смысл |
| --- | --- |
| `10` | Нет специальных функций |
| `11` | Sleep / сон |
| `12` | Ionizer / ионизатор |
| `14` | Mute / quiet / тихий режим |
| `18` | Strong / turbo |
| `1A` | Strong / turbo + ionizer |

`1A` подтверждает, что поле является битовой маской:

```text
0x1A = 0x10 | 0x08 | 0x02
```

## Swing

Состояние swing хранится в `payload[4]`.

Наблюдаемые значения:

| Значение | Смысл |
| --- | --- |
| `18` | Normal/off |
| `1A` | Horizontal swing |

Возможно, это тоже битовая маска, но пока подтвержден только horizontal swing.

## Известные кнопки пульта

| Кнопка пульта | Поле состояния |
| --- | --- |
| Power | `payload[1]` |
| Mode / `РЕЖИМ` | `payload[5]` |
| Fan / `СКОРОСТЬ` | `payload[7]` |
| `+` / `-` | `payload[8]` |
| Turbo / `ТУРБО` | `payload[6] & 0x08` |
| Ionizer, кнопка с листиком | `payload[6] & 0x02` |
| Quiet / `ТИХИЙ` | `payload[6] & 0x04` |
| Sleep / `СОН` | `payload[6] & 0x01` |
| Кнопка horizontal swing | `payload[4]`, наблюдалось `18 <-> 1A` |

## Открытые вопросы

- Точное значение `payload[3]`; наблюдалось `06 <-> 00`, возможно display/light.
- Точное кодирование всех swing-режимов в `payload[4]`.
- Значение `payload[2]`.
- Значение `payload[10..18]` в обычных кадрах состояния.
- Должны ли timestamp-байты команды `payload[15..16]` совпадать с реальным
  локальным временем или они только информационные.
- Отклоняет ли кондиционер какие-либо комбинации специальных функций.

## Заметки по реализации

Для парсера состояния:

1. Ждать байт заголовка `AA`.
2. Читать `LEN`.
3. Читать `LEN` байт.
4. Проверить контрольную сумму через `sum(all bytes before checksum) & 0xFF`.
5. Для `LEN == 0x14` разобрать 19-байтовый payload и опубликовать состояние
   climate.

Для специальных функций:

```cpp
const uint8_t special = payload[6];
const bool sleep = special & 0x01;
const bool ionizer = special & 0x02;
const bool mute = special & 0x04;
const bool turbo = special & 0x08;
```

Базовый marker-бит `0x10` ожидается в обычных кадрах состояния.
